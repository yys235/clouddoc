## 2. 资料来源摘要

### 2.1 飞书官网云文档能力

飞书官网说明「云文档」是在线、多人协作的 Office 套件，包含文档、电子表格、多维表格、幻灯片、思维笔记、知识库、我的文档库和云盘。官网还明确提到云文档支持多人协作、权限隔离、全文搜索、评论、导出、历史记录等基础能力。

关键事实：

- 云文档支持多人共同编辑，官网示例中提到 200 人同时在线编辑、5000 人同时在线阅读。
- 文档权限支持控制是否可分享到组织外、谁可编辑、谁可阅读、谁可导出或创建副本、是否允许他人分享。
- 文档编辑器通过「+」菜单插入丰富内容，通过「=」菜单调整块级格式，通过悬浮菜单调整局部文本格式。
- 官方资料明确提到「流程图或 UML 图」能力：无需下载软件或注册其他网站，可以画流程图、组织架构图等，并支持多人协作共创。

### 2.2 飞书画板开放生态线索

公开生态资料显示，飞书画板不是简单图片，而更接近可编程的原生画板节点：

- 有资料提到飞书原生画板块可作为文档 Block 嵌入。
- 公开工具提到画板相关权限，例如 `board:whiteboard:node`、`board:whiteboard:node:create`、`board:whiteboard:node:read`。
- 公开工具工作流通常是：创建文档或使用已有文档，在文档中添加画板，创建形状节点，再创建连接线，最后截图验证。

这些资料不是飞书官方产品手册，不能直接当成产品承诺，但能证明飞书画板在技术形态上至少具备：

- 文档内嵌

- 形状节点
- 连接线节点
- 样式属性
- 读写权限
- 图表代码导入或自动生成的可能性

## 3. 飞书文档【画板】产品定位

### 3.1 不是独立绘图工具，而是文档上下文中的可视化块

飞书文档画板的核心价值不是替代 Figma、Miro 或专业流程图软件，而是在文档上下文中补足「图形表达」：

- 文档负责叙述、背景、结论和执行事项。
- 画板负责关系、流程、结构、架构和空间化表达。
- 评论、权限、分享、历史记录复用文档体系。
- 画板作为文档块存在，天然归属于某篇文档。

### 3.2 典型场景

- 产品流程图
- 业务流程图
- 组织架构图
- 系统架构图
- UML 图
- 时序图
- 状态流转图
- 会议白板草图
- 需求评审中的方案推演
- AI 根据文档内容自动生成图谱

### 3.3 用户价值

- 不需要离开文档去单独打开第三方绘图软件。
- 图和文字在同一份文档中同步沉淀。
- 权限、评论、分享、搜索、历史记录等能力复用文档基础设施。
- 图形内容可以与文档上下文形成闭环，便于评审、协作和归档。

## 4. 飞书画板界面与功能点拆解

### 4.1 文档内入口

已确认和推导的入口：

- 文档块左侧「+」插入菜单。
- 文档空行输入 `/` 打开命令菜单。
- 菜单中选择「流程图 / UML 图 / 画板」类能力。
- 插入后在文档中形成一个画板块。

### 4.2 文档内嵌显示形态

画板块在文档内应表现为一个可视化块，而不是普通链接：

- 阅读模式下显示画板缩略预览。
- 鼠标悬浮时显示操作按钮，例如打开、编辑、复制、更多。
- 可点击进入大画布编辑或全屏查看。
- 块本身参与文档排版、拖拽、复制、删除和评论。
- 权限继承文档权限，至少阅读者只能查看，编辑者才能修改。

### 4.3 画板编辑界面

根据飞书文档块编辑习惯和白板产品通用结构，画板编辑界面建议拆成：

- 顶部工具栏：选择、拖动画布、文本、形状、连接线、画笔、便签、图片、撤销、重做、缩放。
- 左侧图形库：基础图形、流程图图形、UML 图形、组织架构、箭头线条、模板。
- 中央无限画布：支持平移、缩放、框选、拖拽、吸附、对齐。
- 右侧属性面板：颜色、边框、线型、箭头、字体、层级、大小、位置。
- 底部或右下角视图控制：缩放百分比、适应屏幕、小地图、全屏。

### 4.4 画布基础操作

- 平移画布。
- 缩放画布。
- 框选多个对象。
- 拖拽对象。
- 调整对象大小。
- 复制、粘贴、删除。
- 撤销、重做。
- 对齐、分布。
- 组合、取消组合。
- 置顶、置底、上移一层、下移一层。
- 锁定对象。
- 吸附到网格或对象边缘。

### 4.5 图形对象

核心对象类型：

- 文本
- 矩形
- 圆角矩形
- 圆形 / 椭圆
- 菱形
- 容器 / 分组框
- 便签
- 图片
- 连接线
- 箭头
- 自由画笔
- 分隔线

专业图形类型：

- 流程图开始 / 结束
- 流程处理
- 判断节点
- 数据节点
- 子流程
- UML 类
- UML 生命线
- UML 消息线
- 泳道
- 组织架构节点

### 4.6 连接线能力

飞书画板类产品的关键差异在连接线：

- 连接线可绑定到形状锚点。
- 拖动形状后连接线跟随。
- 支持直线、折线、曲线。
- 支持起点 / 终点箭头。
- 支持在线上添加文本标签。
- 支持自动避让或简化折线。
- 支持连接点吸附。

### 4.7 协作能力

基于飞书云文档的协作体系，画板应复用：

- 多人实时编辑。
- 多人光标或头像位置。
- 评论。
- @ 提及。
- 历史记录。
- 权限控制。
- 分享。
- 复制副本。
- 导出。

### 4.8 AI 与自动生成

公开生态资料中已经出现「Markdown 一键变飞书文档并自动生成架构图画板」的场景。因此 CloudDoc 画板应预留：

- 从 Markdown / Mermaid / PlantUML 生成画板。
- 从文档段落提取流程并生成图。
- 从评论或需求描述更新图。
- MCP / Open API 读取画板结构。
- MCP / Open API 创建和修改画板节点。

## 5. 对 CloudDoc 的设计原则

### 5.1 保持文档块一致性

画板必须是文档块的一种：

- `document_type = doc` 的正文中可插入 `board` block。
- 画板块可被拖拽、复制、删除、评论。
- 阅读模式和编辑模式使用同一套渲染机制，只通过 `editable` 控制可编辑。

### 5.2 画板数据独立存储

画板内容不适合直接塞进文档正文 JSON 的普通文本块中。

建议：

- 文档正文里保存 `board_block`，只包含 `board_id`、标题、预览图、尺寸、更新时间。
- 画板实体存储在独立 `boards` 表。
- 画板节点存储在 `board_nodes` 表或 `boards.content_json` 中。
- V1 可以先用 `boards.content_json`，V2 再拆节点表。

### 5.3 权限继承文档，后续再独立授权

首版规则：

- 画板块继承所在文档权限。
- 文档可读则画板可读。
- 文档可编辑则画板可编辑。
- 分享文档时画板可只读查看。

后续增强：

- 独立画板文档类型。
- 独立画板分享。
- 画板级评论和审计。

## 6. PRD V1：文档内嵌基础画板

### 6.1 版本目标

实现最小可用闭环：

- 能在文档中插入画板块。
- 能打开画板编辑器。
- 能绘制基础流程图。
- 能保存并在文档中显示预览。
- 暂不做多人实时协作。

### 6.2 用户故事

- 作为文档编辑者，我可以在文档中插入一个画板，用于补充流程图。
- 作为文档编辑者，我可以在画板中添加形状、文本和连接线。
- 作为文档读者，我可以在文档中看到画板预览，并打开查看大图。
- 作为文档编辑者，我可以删除、复制、移动画板块。

### 6.3 功能范围

#### 插入能力

- `/画板` 或 `/流程图` 命令。
- 文档块工具菜单中增加「画板」。
- 插入后生成一个 `board` block。

#### 画板编辑器

- 画布平移和缩放。
- 新增文本。
- 新增矩形、圆角矩形、圆形、菱形。
- 新增连接线。
- 拖拽移动对象。
- 调整对象大小。
- 删除对象。
- 撤销 / 重做。
- 保存。

#### 文档内预览

- 显示画板标题。
- 显示缩略预览。
- 显示「打开」「编辑」入口。
- 只读模式下只能打开查看。
- 编辑模式下可进入编辑器。

### 6.4 数据模型

新增 `boards` 表：

- `id`
- `document_id`
- `space_id`
- `owner_id`
- `title`
- `content_json`
- `preview_url`
- `created_at`
- `updated_at`

`content_json` V1 示例：

```json
{
  "version": 1,
  "viewport": { "x": 0, "y": 0, "zoom": 1 },
  "nodes": [
    {
      "id": "node_1",
      "type": "shape",
      "shape": "round_rect",
      "x": 100,
      "y": 100,
      "width": 160,
      "height": 48,
      "text": "开始",
      "style": {
        "fill": "#e8f1ff",
        "stroke": "#2f6fed"
      }
    }
  ],
  "edges": [
    {
      "id": "edge_1",
      "from": "node_1",
      "to": "node_2",
      "type": "polyline",
      "arrow": "end"
    }
  ]
}
```

文档 `board` block 示例：

```json
{
  "type": "board",
  "attrs": {
    "block_id": "block_x",
    "board_id": "board_x",
    "title": "登录流程图",
    "preview_url": "/api/boards/board_x/preview"
  }
}
```

### 6.5 V1 不做

- 多人实时协作。
- AI 生成。
- 原生 UML 模板库。
- 复杂图形库。
- 画板独立分享。
- 画板导出 SVG / PDF。
- 离线编辑。

### 6.6 验收标准

- 可以在文档编辑器中插入画板块。
- 刷新页面后画板块仍存在。
- 打开画板可新增 4 种基础形状和连接线。
- 保存后重新打开，节点位置、文字、颜色、连接关系不丢失。
- 阅读模式下画板不可编辑。
- 分享只读文档时画板只能查看。

## 8. PRD V3：实时协作、AI 画图与开放接口

### 8.1 版本目标

对齐飞书式高级能力：

- 画板支持多人实时协作。
- AI 能根据文档内容生成画板。
- MCP / Open API 能读取和创建画板。
- 画板可以作为独立文档类型存在。

### 8.2 实时协作

- 多用户同时编辑。
- 用户头像和在线光标。
- 对象级锁或 CRDT 合并。
- 冲突处理。
- 实时保存。
- 断线重连。

### 8.3 AI 能力

- 从选中文本生成流程图。
- 从整篇文档生成架构图。
- 从 Mermaid 生成画板。
- 从 PlantUML 生成画板。
- 从画板反向生成文字说明。
- AI 优化布局。
- AI 检查流程断点。

### 8.4 MCP / Open API

读取接口：

- 获取画板元信息。
- 获取画板节点。
- 获取画板连接线。
- 导出画板为 Markdown 描述。
- 导出画板为 Mermaid。

写入接口：

- 创建画板。
- 新增节点。
- 更新节点。
- 删除节点。
- 新增连接线。
- 更新布局。

权限规则：

- API 和 MCP 使用统一 ActorContext。
- 读取遵循文档权限和公开分享权限。
- 写入必须是文档作者、画板作者或有编辑权限的用户。
- 删除必须是画板作者、文档作者或管理员。
- 所有 MCP 写操作进入审计日志。

### 8.5 独立画板文档类型

新增 `document_type = board`：

- 可在文件夹树中独立创建画板。
- 可被文档作为块引用。
- 可被多个文档引用同一个画板。
- 支持独立权限、分享、收藏、最近访问。

### 8.6 验收标准

- 两个用户同时打开同一画板，可以看到对方的编辑。
- AI 可以根据一段流程描述生成可编辑画板。
- MCP 可以读取指定画板并返回结构化 JSON。
- MCP 可以创建一个基础流程图画板。
- 独立画板可以出现在文件夹树中。

## 10. 技术选型建议

### 10.1 渲染层

V1 推荐 SVG + React：

- 对象可控。
- 容易选中、拖拽、编辑。
- 容易导出。
- 方便自动化测试。
- 适合流程图、架构图、组织图。

不建议 V1 直接上纯 Canvas：

- 对象命中测试、文本编辑、评论定位、辅助访问、自动化测试都会更复杂。

### 10.2 状态模型

V1：

- 前端维护完整 `BoardState`。
- 保存时整体写入 `boards.content_json`。
- 撤销重做在前端维护操作栈。

V2：

- 对象级 patch。
- 版本快照。
- 评论锚点绑定对象 ID。

V3：

- CRDT 或操作日志。
- 后端实时广播。
- MCP 使用服务层 API，不直接写 SQL。

## 11. 风险与约束

### 11.1 最大风险：范围膨胀

画板很容易变成「在线 Figma / Miro」。CloudDoc 应坚持文档场景：

- 先流程图。
- 再架构图。
- 最后白板协作。

### 11.2 数据模型风险

如果 V1 把画板内容直接塞进文档正文，会影响后续独立画板、复用、权限和 API。因此必须从第一版就独立 `boards`。

### 11.3 协作复杂度风险

实时协作不要进入 V1。V1 只做单人编辑 + 保存；V2 做评论评审；V3 再做实时协作。

### 11.4 兼容性风险

飞书真实画板可能包含更多内部对象类型。CloudDoc 不需要完全复制，但需要保证：

- 核心流程图可用。
- 文档内嵌体验一致。
- 未来能扩展对象类型。

## 12. 结论

CloudDoc 的画板能力应该走「文档上下文可视化」路线，而不是一开始做通用白板。

三版路线建议：

- V1：文档内嵌基础画板，解决能画、能保存、能查看。
- V2：专业流程图和评审，解决产品研发团队真的可用。
- V3：实时协作、AI 生成和开放接口，形成差异化。

这个路线能最大化复用 CloudDoc 已有文档块、权限、评论、分享、MCP 和文件夹体系，同时避免前期复杂度失控。

## 13. 二次调研依据

### 13.1 官方确认点

飞书官网对云文档编辑器的描述给出几个关键 UI 方向：

- 新文档页面非常简洁，初始只有标题、正文和加号。
- 更多功能隐藏在三种浮动菜单中：`+` 菜单、`=` 区块工具栏、选中文本后的悬浮菜单。
- `+` 菜单用于新增内容，包含图片、任务、网页卡片、投票、流程图或 UML 图等。
- 流程图或 UML 图可以直接在飞书云文档内制作，不需要跳转第三方软件，并支持多人协作共创。
- 左侧目录根据标题生成，强调文档结构和阅读效率。

这说明 CloudDoc 画板入口不应做成沉重的顶部大工具栏，而应延续文档块编辑模式：

- 默认阅读/写作界面保持干净。
- 画板入口藏在块级 `+` / `/` 命令菜单中。
- 画板工具在用户进入画板编辑态时才展开。

### 13.2 公开 API 与生态线索

公开 API 资料显示：

- 飞书新版文档存在「流程图 & UML Block」。
- 绘图类型至少区分「流程图」和「UML 图」。
- Block 视图类型存在卡片视图、预览视图、内联视图。

公开生态资料还显示：

- 飞书画板可被视为文档内的子应用或原生块。
- Mermaid / PlantUML 自动生成画板是实际存在的集成方向。
- 画板相关开放能力可能涉及 `board:whiteboard:node` 这类读写权限。
- 第三方导出工具通常无法完整导出飞书流程图 / UML / 画板，说明这类内容不是普通图片，而是独立子应用数据。

### 13.3 对 CloudDoc 的 UI 设计约束

- 不能把画板做成孤立应用；首屏仍应像文档。
- 不能让文档块显得笨重；画板块在阅读态应轻量、干净、可预览。
- 画板编辑器应是专业工作台，但入口必须克制。
- V1 不做完整白板；先做流程图，避免范围失控。
- 页面样式要专业、紧凑、无过度圆角，符合当前 CloudDoc 已经调整过的视觉方向。

## 14. 改进版 PRD 总览

本次从页面样式和功能点角度拆成 5 个版本：

- V1：文档内嵌画板预览块
- V2：基础流程图编辑器
- V3：飞书式紧凑专业画板工作台
- V4：评审协作与演示模式
- V5：AI 生成、模板市场与开放接口

这 5 个版本不是并列方案，而是连续迭代路线。每一版都必须在上一版稳定后再进入下一版。

## 15. PRD V1：文档内嵌画板预览块

### 15.1 版本目标

先把「画板是文档块」这件事做对。用户在 CloudDoc 文档里插入画板后，应像飞书文档一样，在文档上下文中看到一个轻量、干净、可操作的画板预览块。

### 15.2 页面样式

#### 阅读态

- 画板块宽度跟随文档正文最大宽度。
- 默认不显示厚边框，不做卡片化大阴影。
- 背景使用接近白色的浅灰画布底，和正文区域有轻微分隔。
- 顶部显示一行小标题：画板名称、更新时间、对象数量。
- 中间显示缩略图预览。
- 鼠标悬浮时右上角出现轻量操作区：
  - 打开
  - 复制链接
  - 更多
- 只读模式下不显示编辑入口。
- 加载失败时显示低干扰错误态，而不是整块红色报错。

#### 编辑态

- 文档页面整体布局不跳动。
- 画板块位置、宽度、高度和阅读态一致。
- 悬浮操作区增加「编辑」按钮。
- 支持块级拖拽、复制、删除、评论。
- 光标进入画板块附近时显示左侧块工具按钮，风格与文本块一致。

### 15.3 功能点

- `/画板` 插入空白画板。
- `/流程图` 插入流程图画板。
- `/UML` 插入 UML 画板。
- 块工具菜单中增加「画板」。
- 新建画板时生成 `board` block 和 `boards` 记录。
- 文档保存后刷新，画板块不丢失。
- 画板块可移动到其他文档块之间。
- 画板块可删除，删除前弹窗确认。

### 15.4 验收标准

- 文档内可插入画板块。
- 阅读态和编辑态布局一致。
- 悬浮操作只在鼠标进入画板块时出现。
- 刷新页面后画板块和预览仍存在。
- 删除画板块必须二次确认。

## 16. PRD V2：基础流程图编辑器

### 16.1 版本目标

让用户能完成一张基础流程图：节点、文字、连接线、保存、重新打开不丢失。

### 16.2 页面样式

#### 编辑器整体布局

- 打开画板时进入独立编辑层，可以是全屏 Drawer / 新页面。
- 顶部高度控制在 44px 到 52px，保持专业工具感。
- 左侧工具栏使用垂直窄栏，宽度 44px 到 56px。
- 中央画布使用浅灰网格背景。
- 右侧属性面板默认收起，选中对象后展开。
- 底部右侧显示缩放控制和适应屏幕按钮。

#### 顶部工具栏

左侧：

- 返回文档
- 画板标题
- 保存状态

中间：

- 撤销
- 重做
- 选择
- 手型拖动
- 文本
- 形状
- 连接线

右侧：

- 预览
- 导出
- 保存

### 16.3 功能点

- 画布平移。
- 画布缩放。
- 新增文本节点。
- 新增矩形、圆角矩形、圆形、菱形。
- 新增连接线。
- 连接线可连接到节点。
- 拖动节点时连接线跟随。
- 节点文字可编辑。
- 节点可调整大小。
- 节点可删除。
- 支持撤销 / 重做。
- 支持保存。

### 16.4 验收标准

- 能绘制「开始 -> 判断 -> 处理 -> 结束」流程图。
- 节点和连接线保存后刷新不丢失。
- 拖动节点后连接线仍正确连接。
- 撤销和重做能覆盖新增、移动、删除、改文字。

## 17. PRD V3：飞书式紧凑专业画板工作台

### 17.1 版本目标

把 V2 从「能画」提升到「产品经理、研发、运营可以日常使用」。重点是样式专业化、工具紧凑化、图形库完善。

### 17.2 页面样式

#### 视觉方向

- 整体不要大圆角。
- 背景使用冷灰色工具面板 + 白色画布。
- 选中对象使用蓝色边框和 8 个控制点。
- 对齐辅助线使用高亮蓝或紫色细线。
- 工具按钮采用 32px 正方形，悬浮时浅灰底。
- 工具分组之间用细分割线，不用厚卡片。

#### 图形库面板

- 左侧工具栏点击「形状」后弹出紧凑图形库。
- 图形库宽度约 260px。
- 分组显示：
  - 基础图形
  - 流程图
  - UML
  - 容器
  - 箭头
- 每个图形只显示小图标和短名称。
- 支持搜索图形。

#### 属性面板

- 右侧属性面板宽度约 280px。
- 只在选中对象后显示。
- 内容分组：
  - 位置尺寸
  - 填充
  - 边框
  - 文本
  - 连接线
- 默认只展开最常用项，避免过长。

### 17.3 功能点

- 多选。
- 框选。
- 复制 / 粘贴。
- 删除。
- 组合 / 取消组合。
- 对齐左、右、上、下、中。
- 水平 / 垂直分布。
- 图层置顶 / 置底。
- 网格吸附。
- 对象边缘吸附。
- 复制样式。
- 快捷键：
  - Delete 删除
  - Cmd/Ctrl + Z 撤销
  - Cmd/Ctrl + Shift + Z 重做
  - Cmd/Ctrl + C 复制
  - Cmd/Ctrl + V 粘贴
  - Cmd/Ctrl + D 复制一份

### 17.4 模板功能

- 空白流程图。
- 登录流程图。
- 审批流程图。
- 系统架构图。
- 用户旅程图。
- 组织架构图。

### 17.5 验收标准

- 多选、对齐、分布、复制样式可用。
- 工具面板不遮挡画布核心编辑区域。
- 小屏下右侧属性面板可自动收起。

## 18. PRD V4：评审协作与演示模式

### 18.1 版本目标

围绕文档评审和团队沟通补齐协作能力。这个版本不追求多人实时共同拖节点，而是先解决「看、评、改、定稿」。

### 18.2 页面样式

#### 评论侧栏

- 复用文档评论侧栏风格。
- 评论列表显示在画板右侧。
- 点击评论后画布平滑定位到对应对象或区域。
- 被评论对象显示淡黄色边框或浮标。
- 已解决评论折叠显示。

#### 演示模式

- 全屏展示画板。
- 隐藏编辑工具栏。
- 只保留缩放、上一页/下一页、退出。
- 支持按分组或框选区域生成演示路径。
- 当前演示区域以轻微放大和遮罩突出。

### 18.3 功能点

- 对单个对象评论。
- 对选区评论。
- 评论 @ 人。
- 评论解决 / 重新打开。
- 删除对象时处理评论：
  - 未解决评论标记为「对象已删除」
  - 已解决评论保持归档
- 查看历史版本。
- 手动创建命名版本。
- 恢复历史版本前二次确认。
- 导出 PNG。
- 复制为图片。
- 复制为文档链接。

### 18.4 验收标准

- 用户可以在画板对象上发起评论。
- 点击评论能定位到对象。

- 可以导出 PNG。
- 可以全屏演示画板。

## 21. 推荐 UI 关键决策

### 21.1 文档页不要工具常驻

文档页只显示画板块预览，不常驻画板工具栏。这样和飞书「简洁编辑器 + 浮动菜单」方向一致。

### 21.2 编辑器使用专业工具台

真正进入画板编辑时，才展示完整工具台。这样能同时满足：

- 文档阅读不被打扰。
- 画图时功能足够专业。

### 21.3 先流程图，后白板

V1 到 V3 应先围绕流程图和 UML。自由白板、画笔、贴纸、会中协作放到 V4/V5 之后。

### 21.4 块预览必须轻量

画板块不要像大卡片一样突兀。建议使用：

- 浅灰画布背景
- 细边框
- 小标题
- 悬浮操作
- 无重阴影

### 21.5 保持 CloudDoc 现有视觉方向

当前 CloudDoc 已经明确趋向：

- 紧凑
- 专业
- 少圆角或无圆角
- 文档阅读与编辑一致

画板页面应继续沿用，不要引入明显不同的设计语言。

## 22. 更新后的结论

第一版三阶段 PRD 解决了「做什么」的问题；本次五版本 PRD 进一步解决「页面长什么样、功能怎么铺开」的问题。

推荐采用的最终路线：

1. 先做文档内画板预览块，确保文档体验正确。
2. 再做基础流程图编辑器，确保能画、能保存。
3. 再做专业工作台，让功能真正可用。
4. 再做评论、版本、演示，服务团队评审。
5. 最后做 AI 生成、模板市场、MCP / Open API，形成 CloudDoc 差异化。

这个路线既参考了飞书的文档内子应用思路，也符合 CloudDoc 当前的块文档、权限、评论、MCP 和文件夹体系。
