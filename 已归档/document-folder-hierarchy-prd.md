# 文档层级与文件夹专项 PRD（Folder 独立建模版）

## 1. 背景

当前 CloudDoc 的文档在产品表现上仍然是扁平列表：

- “我的文档”页按线性列表展示
- 文档之间没有清晰的层级归属
- 用户无法用文件夹组织内容

同时，你已经明确产品后续目标是：

- 通用云盘
- 通用文档系统
- 后续会接入更多对象类型

在这个前提下，文件夹不适合继续作为“特殊文档类型”塞进 `documents` 表。文件夹应该单独建模，成为内容树中的正式容器对象。

---

## 2. 目标

### 2.1 产品目标

- 引入正式文件夹概念
- 支持空间内的树状层级管理
- 支持文档归档到文件夹
- 为后续接入 PDF、图片、视频、表格、白板等对象打基础

### 2.2 工程目标

- 文件夹单独成表，不再混入 `documents`
- 文档和文件夹通过统一树关系组织
- 当前已有文档统一归档到每个空间下的 `/newdoc`
- 不破坏现有评论、分享、收藏、通知、权限主链路

---

## 3. 原始需求与扩展定义

### 3.1 原始需求

- 当前文档是扁平化的
- 需要层级或文件夹概念
- 引入文件夹概念
- 当前已有文档统一归纳到 `/newdoc` 文件夹下面

### 3.2 扩展后的明确需求

落实为可开发需求后，定义如下：

- 文件夹是空间内的正式容器
- 文件夹支持嵌套子文件夹
- 文档挂在空间根目录或某个文件夹下
- 文件夹本身不进入文档编辑器
- 当前所有历史文档都迁移到各自空间根目录下的 `newdoc` 文件夹
- 新建文档时必须允许确认所属位置，默认使用当前目录，并支持改选空间根目录、已有文件夹，或新建文件夹后创建到新文件夹
- 上传 PDF 时也允许指定所属文件夹
- 新建文件夹时允许指定父文件夹
- 左侧文件夹目录树支持拖拽文档或文件夹更换位置
- 左侧文件夹目录树支持将文档或文件夹拖入目标文件夹

### 3.3 本期不做

- 文件夹级分享
- 文件夹级评论
- 文件夹级收藏
- 文件夹级模板
- 文件夹递归删除
- 跨空间拖拽移动

---

## 4. 设计结论

## 4.1 Folder 单独建模

本期采用：

- `folders` 独立表

而不是：

- `documents.document_type = folder`

原因：

- 文件夹不是文档，不应混在文档表里
- 后续如果扩成通用云盘，文件夹会成为独立基础设施
- 后续新类型对象接入时，不需要继续往 `documents` 表里塞非文档对象

## 4.2 树模型

本期采用“两棵对象，共一棵树”的方式：

- `folders` 自身形成文件夹树
- `documents` 挂到 `folders` 或空间根目录下

即：

- 文件夹可以有子文件夹
- 文件夹可以包含文档
- 文档不能拥有子节点

### 统一树视角

在产品表现上，前端看到的是一棵统一树：

- `space root`
  - `folder`
    - `folder`
      - `doc`
      - `pdf`
    - `doc`
  - `doc`

但底层存储上：

- 文件夹节点来自 `folders`
- 文档节点来自 `documents`

---

## 5. 数据模型

## 5.1 新增 folders 表

建议新增：

- `folders`

字段如下：

- `id`
- `space_id`
- `parent_folder_id`
- `creator_id`
- `owner_id`
- `title`
- `visibility`
- `icon`
- `sort_order`
- `is_deleted`
- `created_at`
- `updated_at`

### 字段说明

- `space_id`
  - 所属空间
- `parent_folder_id`
  - 父文件夹 ID
  - `null` 表示空间根目录
- `visibility`
  - 首版可与文档保持一致，仅支持：
    - `private`
    - `public`
  - 但本期不做复杂继承
- `sort_order`
  - 为后续手动排序预留
- `is_deleted`
  - 与文档保持一致，支持软删除

## 5.2 调整 documents 表

当前 `documents` 表已有：

- `space_id`
- `parent_id`

本期建议调整为更清晰的语义：

- 新增 `folder_id`
- 后续逐步废弃 `parent_id`

### documents 表新增/调整字段

- `folder_id UUID NULL REFERENCES folders(id)`

语义：

- `null` 表示挂在空间根目录
- 非空表示文档位于某个文件夹下

### 关于现有 `parent_id`

因为当前线上和代码里已使用 `parent_id`，本期迁移建议分两步：

#### 过渡期

- 保留 `parent_id`
- 新增 `folder_id`
- 新逻辑统一使用 `folder_id`
- 老逻辑不再继续扩展 `parent_id`

#### 收敛期

- 数据迁移完成后
- 删除 `parent_id` 的业务依赖
- 最终再决定是否删字段

---

## 6. 层级规则

## 6.1 文件夹树规则

- 文件夹可有子文件夹
- 文件夹可包含文档
- 文档不可作为父节点
- 不允许循环父子关系

## 6.2 空间根目录规则

空间根目录不是一条真实记录，而是虚拟根节点。

即：

- `parent_folder_id = null` 的文件夹在空间根目录
- `folder_id = null` 的文档也在空间根目录

---

## 7. 历史数据迁移

## 7.1 迁移目标

把当前已有文档统一迁移到各空间根目录下的：

- `newdoc`

文件夹内。

## 7.2 迁移规则

对每个空间：

1. 检查根目录下是否已有 `newdoc`
2. 若无，则创建：
   - `title = 'newdoc'`
   - `parent_folder_id = null`
3. 将该空间下当前所有历史文档统一移动到：
   - `folder_id = newdoc.id`

### 历史文档判定

因为目前没有正式文件夹系统，本期定义“历史根级文档”为：

- `documents.space_id = 当前空间`
- `documents.is_deleted = false`
- 当前没有正式 `folder_id`

### 幂等要求

- 重复执行迁移，不应重复创建 `newdoc`
- 已迁移文档不应重复移动

---

## 8. 页面与交互设计

## 8.1 我的文档页

“我的文档”页从扁平列表升级为树状浏览页。

首版建议：

- 左侧：文件夹树
- 右侧：当前文件夹内容列表

如果当前实现节奏更稳，也可以先做：

- 单页双栏
  - 左边树
  - 右边当前目录内容

## 8.2 文件夹页

点击文件夹后，进入文件夹详情页。

展示内容：

- 子文件夹
- 子文档
- 面包屑路径
- 新建文件夹
- 新建文档
- 上传 PDF

## 8.3 文档页面包屑

文档页顶部面包屑从当前：

- `空间 / 云文档`

升级为：

- `空间 / newdoc / 子文件夹 / 文档标题`

要求：

- 每一级可点击
- 面包屑路径要同时支持文档和文件夹

## 8.4 文件夹操作

文件夹首版支持：

- 新建
- 重命名
- 删除（仅空文件夹）
- 移动文档进入当前文件夹
- 新建文档时弹出位置确认窗口，避免用户误把文档创建到空间根目录或错误文件夹

## 9. 接口设计

## 9.1 文件夹接口

新增：

- `POST /api/folders`
- `GET /api/folders/{id}`
- `PATCH /api/folders/{id}`
- `DELETE /api/folders/{id}`

### 创建文件夹请求体

```json
{
  "title": "方案库",
  "space_id": "space-id",
  "parent_folder_id": null
}
```

## 9.2 文件夹内容查询

新增：

- `GET /api/folders/{id}/children`

返回：

- `folders`
- `documents`

### 返回建议

前端应拿到统一结构，例如：

```json
{
  "folder": { "...": "..." },
  "children": [
    { "node_type": "folder", "...": "..." },
    { "node_type": "document", "...": "..." }
  ]
}
```

## 9.3 空间根目录查询

新增：

- `GET /api/spaces/{space_id}/root-children`

返回：

- 根目录下所有文件夹
- 根目录下所有文档

## 9.4 文档移动

新增：

- `POST /api/documents/{id}/move`

请求体：

```json
{
  "folder_id": "target-folder-id"
}
```

规则：

- `folder_id = null` 表示移回空间根目录
- 只能移动到本空间内合法文件夹
- 拖拽到文件夹行中间表示移动进该文件夹
- 拖拽到任意节点上方或下方表示移动到该节点前后，并同步同级排序

## 9.5 文件夹祖先链

新增：

- `GET /api/folders/{id}/ancestors`

用于：

- 面包屑展示
- 路径跳转

## 9.6 文档祖先链

新增：

- `GET /api/documents/{id}/ancestors`

返回：

- space
- 所属文件夹祖先链
- 当前文档

---

## 10. 权限规则

首版不单独发明复杂的文件夹继承权限系统。

### 基础规则

- 查看文件夹：需要有对应空间访问权限
- 新建文件夹：需要空间写权限
- 重命名文件夹：需要文件夹 owner 或空间管理权限
- 删除文件夹：需要文件夹 owner 或空间管理权限
- 移动文档到文件夹：需要文档管理权限 + 目标文件夹写权限

### 首版不做

- 文件夹继承权限到子文档
- 文件夹级 ACL 编辑面板

## 11. 前端实现建议

## 11.1 统一 TreeNode 结构

虽然后端分表，前端仍建议统一抽象成：

- `TreeNode`

字段：

- `id`
- `nodeType`：`folder | doc | pdf`
- `title`
- `spaceId`
- `parentFolderId`
- `updatedAt`
- `visibility`

## 11.2 页面建议

- `/documents`
  - 文档树首页
- `/folders/[folderId]`
  - 文件夹详情页
- `/docs/[docId]`
  - 文档详情页

## 11.3 首版交互建议

先做稳的版本，不一开始上复杂拖拽树。

首版建议支持：

- 展开/收起
- 新建文件夹
- 新建文档
- 上传 PDF
- 移动到文件夹（弹窗选择）
- 左侧目录树拖拽移动到文件夹
- 左侧目录树拖拽调整同级顺序
- 面包屑导航

后续再做：

- 批量移动
- 跨空间拖拽移动

## 13. 测试计划

## 13.1 数据迁移

- 每个空间只创建一个 `newdoc`
- 历史文档被正确迁入 `newdoc`
- 重复执行迁移不出错

## 13.2 文件夹 CRUD

- 根目录可创建文件夹
- 子文件夹可创建
- 文件夹可重命名
- 非空文件夹不可删除
- 空文件夹可删除

## 13.3 文档归档

- 可在文件夹中创建文档
- 可在文件夹中上传 PDF
- 可将文档移入文件夹
- 可将文档移回根目录
- 可在左侧目录树中将文档拖入其他文件夹
- 可在左侧目录树中拖动文档到同级节点前后调整顺序
- 可在左侧目录树中拖动文件夹到同级节点前后调整顺序

## 13.4 前端展示

- 树结构正确
- 面包屑正确
- 文件夹页和文档页路径一致
- 根目录与文件夹页展示逻辑一致
- 拖拽时显示插入线或目标文件夹高亮
- 拖拽完成后目录树刷新并保持数据一致

---

## 14. 最终结论

考虑到产品后续明确会走向通用云盘/通用文档系统，本期不再建议把文件夹伪装成文档。

本期正式采用：

- `folders` 独立表
- `documents.folder_id`
- 文件夹树 + 文档挂载

并通过一次性迁移，把当前历史文档统一归档到：

- 每个空间根目录下的 `/newdoc`

这样既满足当前“文件夹层级”需求，也为后续扩展到更多内容类型留出更合理的结构空间。
