# 文档层级与文件夹专项 PRD（Folder 独立建模版）

## 1. 背景

当前 CloudDoc 的文档在产品表现上仍然是扁平列表：

- “我的文档”页按线性列表展示
- 文档之间没有清晰的层级归属
- 用户无法用文件夹组织内容

同时，你已经明确产品后续目标是：

- 通用云盘
- 通用文档系统
- 后续会接入更多对象类型

在这个前提下，文件夹不适合继续作为“特殊文档类型”塞进 `documents` 表。文件夹应该单独建模，成为内容树中的正式容器对象。

---

## 2. 目标

### 2.1 产品目标

- 引入正式文件夹概念
- 支持空间内的树状层级管理
- 支持文档归档到文件夹
- 为后续接入 PDF、图片、视频、表格、白板等对象打基础

### 2.2 工程目标

- 文件夹单独成表，不再混入 `documents`
- 文档和文件夹通过统一树关系组织
- 当前已有文档统一归档到每个空间下的 `/newdoc`
- 不破坏现有评论、分享、收藏、通知、权限主链路

---

## 3. 原始需求与扩展定义

### 3.1 原始需求

- 当前文档是扁平化的
- 需要层级或文件夹概念
- 引入文件夹概念
- 当前已有文档统一归纳到 `/newdoc` 文件夹下面

### 3.2 扩展后的明确需求

落实为可开发需求后，定义如下：

- 文件夹是空间内的正式容器
- 文件夹支持嵌套子文件夹
- 文档挂在空间根目录或某个文件夹下
- 文件夹本身不进入文档编辑器
- 当前所有历史文档都迁移到各自空间根目录下的 `newdoc` 文件夹
- 新建文档时必须允许确认所属位置，默认使用当前目录，并支持改选空间根目录、已有文件夹，或新建文件夹后创建到新文件夹
- 上传 PDF 时也允许指定所属文件夹
- 新建文件夹时允许指定父文件夹
- 左侧文件夹目录树支持拖拽文档或文件夹更换位置
- 左侧文件夹目录树支持将文档或文件夹拖入目标文件夹

### 3.3 本期不做

- 文件夹级分享
- 文件夹级评论
- 文件夹级收藏
- 文件夹级模板
- 文件夹递归删除
- 跨空间拖拽移动

---

## 4. 设计结论

## 4.1 Folder 单独建模

本期采用：

- `folders` 独立表

而不是：

- `documents.document_type = folder`

原因：

- 文件夹不是文档，不应混在文档表里
- 后续如果扩成通用云盘，文件夹会成为独立基础设施
- 后续新类型对象接入时，不需要继续往 `documents` 表里塞非文档对象

## 4.2 树模型

本期采用“两棵对象，共一棵树”的方式：

- `folders` 自身形成文件夹树
- `documents` 挂到 `folders` 或空间根目录下

即：

- 文件夹可以有子文件夹
- 文件夹可以包含文档
- 文档不能拥有子节点

### 统一树视角

在产品表现上，前端看到的是一棵统一树：

- `space root`
  - `folder`
    - `folder`
      - `doc`
      - `pdf`
    - `doc`
  - `doc`

但底层存储上：

- 文件夹节点来自 `folders`
- 文档节点来自 `documents`

---

## 5. 数据模型

## 5.1 新增 folders 表

建议新增：

- `folders`

字段如下：

- `id`
- `space_id`
- `parent_folder_id`
- `creator_id`
- `owner_id`
- `title`
- `visibility`
- `icon`
- `sort_order`
- `is_deleted`
- `created_at`
- `updated_at`

### 字段说明

- `space_id`
  - 所属空间
- `parent_folder_id`
  - 父文件夹 ID
  - `null` 表示空间根目录
- `visibility`
  - 首版可与文档保持一致，仅支持：
    - `private`
    - `public`
  - 但本期不做复杂继承
- `sort_order`
  - 为后续手动排序预留
- `is_deleted`
  - 与文档保持一致，支持软删除

## 5.2 调整 documents 表

当前 `documents` 表已有：

- `space_id`
- `parent_id`

本期建议调整为更清晰的语义：

- 新增 `folder_id`
- 后续逐步废弃 `parent_id`

### documents 表新增/调整字段

- `folder_id UUID NULL REFERENCES folders(id)`

语义：

- `null` 表示挂在空间根目录
- 非空表示文档位于某个文件夹下

### 关于现有 `parent_id`

因为当前线上和代码里已使用 `parent_id`，本期迁移建议分两步：

#### 过渡期

- 保留 `parent_id`
- 新增 `folder_id`
- 新逻辑统一使用 `folder_id`
- 老逻辑不再继续扩展 `parent_id`

#### 收敛期

- 数据迁移完成后
- 删除 `parent_id` 的业务依赖
- 最终再决定是否删字段

---

## 6. 层级规则

## 6.1 文件夹树规则

- 文件夹可有子文件夹
- 文件夹可包含文档
- 文档不可作为父节点
- 不允许循环父子关系

## 6.2 空间根目录规则

空间根目录不是一条真实记录，而是虚拟根节点。

即：

- `parent_folder_id = null` 的文件夹在空间根目录
- `folder_id = null` 的文档也在空间根目录

---

## 7. 历史数据迁移

## 7.1 迁移目标

把当前已有文档统一迁移到各空间根目录下的：

- `newdoc`

文件夹内。

## 7.2 迁移规则

对每个空间：

1. 检查根目录下是否已有 `newdoc`
2. 若无，则创建：
   - `title = 'newdoc'`
   - `parent_folder_id = null`
3. 将该空间下当前所有历史文档统一移动到：
   - `folder_id = newdoc.id`

### 历史文档判定

因为目前没有正式文件夹系统，本期定义“历史根级文档”为：

- `documents.space_id = 当前空间`
- `documents.is_deleted = false`
- 当前没有正式 `folder_id`

### 幂等要求

- 重复执行迁移，不应重复创建 `newdoc`
- 已迁移文档不应重复移动

## 8. 页面与交互设计

## 8.1 我的文档页

“我的文档”页从扁平列表升级为树状浏览页。

首版建议：

- 左侧：文件夹树
- 右侧：当前文件夹内容列表

如果当前实现节奏更稳，也可以先做：

- 单页双栏
  - 左边树
  - 右边当前目录内容

## 8.2 文件夹页

点击文件夹后，进入文件夹详情页。

展示内容：

- 子文件夹
- 子文档
- 面包屑路径
- 新建文件夹
- 新建文档
- 上传 PDF

## 8.3 文档页面包屑

文档页顶部面包屑从当前：

- `空间 / 云文档`

升级为：

- `空间 / newdoc / 子文件夹 / 文档标题`

要求：

- 每一级可点击
- 面包屑路径要同时支持文档和文件夹

## 8.4 文件夹操作

文件夹首版支持：

- 新建
- 重命名
- 删除（仅空文件夹）
- 移动文档进入当前文件夹
- 新建文档时弹出位置确认窗口，避免用户误把文档创建到空间根目录或错误文件夹

## 9. 接口设计

## 9.1 文件夹接口

新增：

- `POST /api/folders`
- `GET /api/folders/{id}`
- `PATCH /api/folders/{id}`
- `DELETE /api/folders/{id}`

### 创建文件夹请求体

```json
{
  "title": "方案库",
  "space_id": "space-id",
  "parent_folder_id": null
}
```

## 9.2 文件夹内容查询

新增：

- `GET /api/folders/{id}/children`

返回：

- `folders`
- `documents`

### 返回建议

前端应拿到统一结构，例如：

```json
{
  "folder": { "...": "..." },
  "children": [
    { "node_type": "folder", "...": "..." },
    { "node_type": "document", "...": "..." }
  ]
}
```

## 9.3 空间根目录查询

新增：

- `GET /api/spaces/{space_id}/root-children`

返回：

- 根目录下所有文件夹
- 根目录下所有文档

## 9.4 文档移动

新增：

- `POST /api/documents/{id}/move`

请求体：

```json
{
  "folder_id": "target-folder-id"
}
```

规则：

- `folder_id = null` 表示移回空间根目录
- 只能移动到本空间内合法文件夹
- 拖拽到文件夹行中间表示移动进该文件夹
- 拖拽到任意节点上方或下方表示移动到该节点前后，并同步同级排序

## 9.5 文件夹祖先链

新增：

- `GET /api/folders/{id}/ancestors`

用于：

- 面包屑展示
- 路径跳转

## 9.6 文档祖先链

新增：

- `GET /api/documents/{id}/ancestors`

返回：

- space
- 所属文件夹祖先链
- 当前文档

---

## 10. 权限规则

首版不单独发明复杂的文件夹继承权限系统。

### 基础规则

- 查看文件夹：需要有对应空间访问权限
- 新建文件夹：需要空间写权限
- 重命名文件夹：需要文件夹 owner 或空间管理权限
- 删除文件夹：需要文件夹 owner 或空间管理权限
- 移动文档到文件夹：需要文档管理权限 + 目标文件夹写权限

### 首版不做

- 文件夹继承权限到子文档
- 文件夹级 ACL 编辑面板

## 12. 开发计划

## Phase 1：数据与迁移

- 新增 `folders` 表
- `documents` 新增 `folder_id`
- 迁移脚本：创建 `newdoc` 并收拢历史文档
- 根目录与文件夹内容查询接口

## Phase 2：前端树状浏览

- 我的文档页改成树状/目录式浏览
- 文件夹页
- 面包屑祖先链

## Phase 3：文件夹管理

- 新建文件夹
- 重命名文件夹
- 删除空文件夹
- 文档移动
- PDF 上传到文件夹

## Phase 4：增强

- 树拖拽
- 排序
- 批量移动
- 文件夹级权限继承

## Phase 5：文档树更多操作菜单

参考飞书云文档左侧文件树的交互方式，在文档树节点右侧增加“更多操作”入口，让用户不必进入文档详情页或右侧列表，也可以直接完成高频文件管理操作。

### 12.1 背景问题

当前 CloudDoc 已具备文件夹树、文档移动、拖拽排序和基础文件夹管理能力，但文件管理操作仍分散在右侧详情区、文档详情页或弹窗入口中。

主要问题：

- 用户在左侧文档树中看到目标文档后，无法直接对该节点执行分享、复制链接、移动、重命名、删除等操作。
- 文件树已经成为主要导航入口，但还不是完整的内容管理入口。
- 拖拽适合调整位置，但不适合所有用户，也不适合触屏或精确移动到复杂目录。
- 批量管理和单节点管理的入口不清晰。
- 文档、PDF、文件夹后续都会进入同一棵树，需要统一的更多操作模型。

### 12.2 产品目标

- 左侧文档树支持节点级快捷操作。
- 文档、PDF、文件夹在树中使用一致的更多操作体验。
- 右键菜单和 `...` 更多按钮共用同一套菜单能力。
- 所有危险操作必须二次确认。
- 菜单操作后通过本地状态和 SSE 局部更新文件树，不做整页刷新。
- 权限不足时菜单项隐藏或禁用，并给出明确原因。

### 12.3 适用对象

首期支持：

- 普通文档 `doc`
- PDF 文档 `pdf`
- 文件夹 `folder`

后续扩展：

- 表格、白板、多维表格等新文档类型
- LLM Wiki 工作区文件夹
- 外部集成生成的文档

### 12.4 菜单入口

每个文档树节点支持两个入口：

- 鼠标悬浮节点时，在节点右侧显示 `...` 按钮。
- 用户右键点击节点时，打开同一个上下文菜单。

显示规则：

- 默认状态不显示 `...`，保持文件树干净。
- 当前 hover 节点显示 `...`。
- 当前选中节点显示 `...`。
- 菜单打开时节点保持浅灰选中底色。
- 节点标题很长时仍要保留右侧 `...` 操作区，标题使用省略号截断。
- `...` 按钮不应改变树节点高度。

### 12.5 菜单视觉规范

菜单参考截图中的紧凑样式：

- 宽度建议 `180px - 220px`。
- 行高建议 `32px`。
- 图标列宽固定，文字左对齐。
- 分组之间使用浅色分割线。
- 菜单背景为白色，边框为浅灰，阴影轻量。
- 菜单应自动避让浏览器边界，靠近右边界时向左展开，靠近底部时向上展开。
- 不使用大圆角，保持当前专业化 UI 风格。

### 12.6 文档节点菜单项

普通文档和 PDF 文档首期菜单：

- 在新标签页打开
- 分享
- 复制链接
- 创建副本
- 移动到
- 添加快捷方式到
- 添加到置顶
- 收藏
- 转移所有权
- 重命名
- 设置密级
- 删除

差异规则：

- PDF 不支持编辑类操作，但支持打开、分享、复制链接、移动、置顶、收藏、重命名、密级、删除。
- 私有文档分享入口可用，但需要进入分享弹窗配置独立分享链接。
- 公开文档复制链接复制原文链接；私有文档如果没有分享链接，复制链接应提示用户先创建分享链接或复制原文内部链接。

### 12.7 文件夹节点菜单项

文件夹首期菜单：

- 打开
- 在新标签页打开
- 新建文档
- 新建文件夹
- 上传 PDF
- 复制链接
- 移动到
- 添加快捷方式到
- 添加到置顶
- 收藏
- 重命名
- 设置密级
- 删除

删除规则：

- 空文件夹可以删除。
- 非空文件夹首期不允许删除，提示“请先移动或删除文件夹内内容”。
- 后续可扩展“删除文件夹及全部内容”，但必须使用强确认弹窗，并明确列出影响范围。

### 12.8 操作行为定义

在新标签页打开：

- 文档打开 `/docs/{docId}`。
- 文件夹打开 `/folders/{folderId}`。
- 遵循用户个人配置中的默认打开方式，但菜单项强制新标签页。

分享：

- 打开现有文档权限/分享弹窗。
- 仅文档支持；文件夹分享作为后续权限体系增强项。

复制链接：

- 复制当前对象链接到剪贴板。
- 成功后显示轻量 toast。
- 失败时显示可手动复制的链接弹窗。

创建副本：

- 仅文档支持。
- 默认在原文档同一文件夹下创建。
- 新文档标题默认 `{原标题} 副本`。
- PDF 创建副本复制文件引用或复制文件对象，具体取决于后端存储实现。

移动到：

- 打开目录树选择弹窗。
- 默认选中当前父文件夹。
- 禁止移动到自身或自身子文件夹。
- 移动完成后左侧树和右侧目录内容局部更新。

添加快捷方式到：

- 后续功能。首期可展示禁用态或不展示。
- 快捷方式不是复制文档，而是创建指向原文档的引用节点。
- 快捷方式删除不删除原文档。

添加到置顶：

- 将节点在当前用户视角置顶。
- 置顶是个人配置，不改变团队其他成员排序。
- 同一文件夹内置顶节点显示在普通节点前。

收藏：

- 收藏当前文档或文件夹。
- 再次打开菜单时显示“取消收藏”。
- 收藏状态属于当前用户。

转移所有权：

- 仅文档所有者或管理员可见。
- 打开所有权转移弹窗。
- 转移前必须二次确认。

重命名：

- 推荐首期使用弹窗重命名，后续可做树内 inline rename。
- 名称不能为空。
- 同一文件夹下是否允许重名按现有产品策略执行；若允许重名，需要通过图标和时间区分。

设置密级：

- 打开权限/安全设置弹窗。
- 首期如果密级模型未完整实现，可进入权限/分享弹窗的安全设置 tab。

删除：

- 所有删除都必须弹确认窗口。
- 删除文档进入回收站。
- 删除文件夹按文件夹删除规则执行。
- 删除成功后通过 SSE 和本地 reducer 移除树节点。

### 12.9 权限规则

菜单项需要根据权限动态显示：

- `can_view`：可打开、复制内部链接。
- `can_share`：可分享。
- `can_edit`：可重命名、移动、创建副本、设置密级。
- `can_delete`：可删除。
- `is_owner`：可转移所有权。
- `is_admin`：可查看和管理更多高级项。

推荐策略：

- 用户完全无权限的节点不应出现在树中。
- 可见但不可执行的操作优先隐藏，避免菜单拥挤。
- 对用户可能期待但权限不足的关键操作可置灰，例如删除、分享。

## 14. 最终结论

考虑到产品后续明确会走向通用云盘/通用文档系统，本期不再建议把文件夹伪装成文档。

本期正式采用：

- `folders` 独立表
- `documents.folder_id`
- 文件夹树 + 文档挂载

并通过一次性迁移，把当前历史文档统一归档到：

- 每个空间根目录下的 `/newdoc`

这样既满足当前“文件夹层级”需求，也为后续扩展到更多内容类型留出更合理的结构空间。
