# 云文档功能设计与 UI 设计详细文档

## 1. 文档目标

本文档用于单独定义在线云文档产品的：
- 功能设计
- UI 设计
- 页面结构
- 交互规则
- 扩展性设计

设计原则参考飞书云文档的产品形态与功能组织方式，但结合本项目当前阶段目标进行裁剪：
- V1 不做实时协同编辑
- V1 只做云文档
- V1 不做多维表格、电子表格、白板、幻灯片
- 底层预留多文档类型支持能力

## 2. 产品定位

产品定位为一款面向个人和团队的在线云文档系统，提供文档创作、文档管理、知识沉淀、分享、权限控制、历史版本、模板、搜索等完整能力。

首版重点不是“多人同时编辑”，而是先实现：
- 好用的文档编辑体验
- 完整的文档管理闭环
- 清晰的知识空间结构
- 平台化能力沉淀

## 3. 对标飞书云文档的能力范围

参考飞书云文档公开能力，云文档产品核心能力可拆为以下几层：
- 内容创作：文档编辑、块编辑、格式化、插入内容块
- 内容组织：个人空间、共享空间、目录树、知识库
- 内容管理：收藏、最近访问、回收站、模板、导入导出
- 内容控制：权限、分享、版本、审阅、历史恢复
- 内容触达：搜索、链接分享、嵌入、消息预览

本项目 V1 对这些能力的取舍如下：

### V1 保留

- 文档编辑
- 文档树与空间管理
- 模板
- 收藏、最近访问、回收站
- 分享与权限
- 历史版本
- 搜索
- 导入导出

### V1 延后

- 实时协同编辑
- 评论讨论流
- 修订模式
- 动态通知
- 消息、会议、日历联动
- 多文档类型落地能力

### V1 必须预留

- `document_type`
- 多类型渲染器注册机制
- 多类型内容 schema 扩展机制
- 通用平台能力复用机制

### 内容模型原则

- 底层采用结构化节点/块模型
- 表现层采用连续文档式视觉和阅读体验
- 交互层支持块级插入、块级选择和块级排序
- 不采用强卡片化、强积木化的 Notion 式表现
- 整体目标更接近飞书云文档的“弱块化表现、强结构化底层”

## 4. 产品信息架构

建议整体采用“左侧导航 + 中间内容区 + 顶部全局工具栏”的典型文档系统布局。

### 4.1 一级导航结构

- 工作台
- 最近访问
- 我的文档
- 团队空间
- 收藏
- 模板中心
- 回收站
- 设置

### 4.2 文档域模型

系统中的核心对象包括：
- 用户
- 组织
- 空间
- 文档
- 模板
- 分享链接
- 版本
- 标签

### 4.3 内容组织层级

建议层级如下：
- 组织
- 空间
- 文件夹/页面树
- 文档

说明：
- 空间用于承载知识归属
- 文件夹和页面树用于承载内容结构
- 文档是最基础的内容单元

## 5. 功能设计

## 5.1 用户与组织

### 功能目标

支持个人使用和团队使用两种模式。

### 功能点

- 邮箱/手机号注册登录
- 创建组织
- 加入组织
- 成员管理
- 角色管理
- 用户个人资料

### 角色定义

- 所有者：管理组织与空间
- 管理员：管理成员、部分内容与权限
- 成员：创建与编辑授权内文档
- 访客：仅访问被授权文档

### UI 设计

- 登录页采用简洁双栏布局
- 左侧展示产品价值点
- 右侧为登录/注册表单
- 注册完成后引导创建个人空间和首个团队空间

## 5.2 工作台

### 功能目标

帮助用户快速进入常用内容，而不是把首页做成纯文件列表。

### 功能点

- 最近访问
- 收藏文档
- 我创建的文档
- 与我共享的文档
- 常用模板
- 空间快捷入口

### UI 设计

工作台首页建议分为四个区域：
- 顶部：全局搜索、新建按钮
- 左侧：导航栏
- 中间：最近访问、常用模板、推荐入口
- 右侧：空间摘要、快捷操作

### 交互说明

- 用户首次进入时，优先展示模板和引导
- 老用户优先展示最近访问和收藏
- 新建按钮固定在顶部高频区域

## 5.3 空间与知识库

### 功能目标

支持用“空间”来承载文档归属，用“目录树”来承载知识组织。

### 功能点

- 个人空间
- 团队空间
- 空间成员可见性配置
- 文件夹管理
- 页面树
- 文档移动
- 文档复制
- 文档排序
- 标签分类

### UI 设计

空间页建议采用三栏布局：
- 左栏：空间切换、目录树
- 中栏：当前文件夹/页面列表
- 顶栏：新建、筛选、排序、搜索

### 交互规则

- 左侧目录树支持展开/收起
- 支持拖拽调整层级
- 鼠标悬停文档项时显示快捷操作
- 新建按钮支持直接在当前目录下创建文档

## 5.4 文档编辑器

### 功能目标

提供接近飞书云文档的结构化富文本编辑体验，在视觉上保持连续文档感，在交互上支持块级操作，同时兼顾传统长文档场景。

### 核心功能点

- 标题与正文
- 标题层级 H1-H3
- 段落
- 加粗、斜体、下划线、删除线
- 行内代码
- 无序列表
- 有序列表
- 任务列表
- 引用块
- 代码块
- 分割线
- 表格块
- 图片块
- 超链接
- 文档大纲
- 快捷命令 `/`
- 自动保存

### 编辑器布局

建议采用如下布局：
- 顶部：文档标题、收藏、分享、权限、更多操作
- 左侧：可折叠目录树或文档大纲
- 中间：编辑区
- 右侧：属性面板或版本/分享面板入口

### 编辑器交互规则

- 光标在空行输入 `/` 呼出块菜单
- 选中文本后出现悬浮格式工具栏
- 标题自动生成大纲
- 插入图片时支持拖拽上传和粘贴上传
- 表格块首版支持基础插入，不做电子表格能力
- 自动保存需提供“已保存/保存中”状态提示

### 编辑器表现原则

- 页面视觉上应像一篇连续文档，而不是一个个明显分裂的卡片块
- 段落之间保留自然排版间距，不强化每个块的边框和容器感
- 块级能力主要通过光标、悬浮操作、插入菜单、拖拽手柄等方式体现
- 标题、段落、列表、图片、代码块等是结构节点，但不需要在默认态高亮“块感”

### 块菜单建议项

- 文本
- 一级标题
- 二级标题
- 三级标题
- 无序列表
- 有序列表
- 任务列表
- 引用
- 代码块
- 分割线
- 图片
- 表格
- 模板片段

### UI 风格建议

- 编辑区居中，宽度控制在适合长文阅读的范围
- 工具栏不宜过重，避免像传统 Office
- 更接近“轻文档 + 内容块”的体验
- 字体、留白、段间距要优先服务长时间阅读

补充说明：
- 这里的“内容块”是底层结构和交互单元，不代表视觉上做成 Notion 式卡片堆叠
- 整体观感应更贴近飞书云文档或现代在线文档，而不是页面搭建器

## 5.5 文档管理

### 功能点

- 新建文档
- 复制文档
- 删除文档
- 回收站恢复
- 移动文档
- 收藏文档
- 文档置顶
- 最近访问
- 文档封面
- 文档图标
- 文档摘要

### 交互规则

- 文档默认自动命名为“未命名文档”
- 删除后进入回收站
- 回收站支持恢复和彻底删除
- 复制文档时默认带上“副本”后缀
- 移动文档时提供目录树选择器

### UI 设计

文档列表项建议展示：
- 图标
- 标题
- 更新时间
- 所属空间/目录
- 创建者
- 快捷操作按钮

## 5.6 模板中心

### 功能目标

提升首版产品的完成度和开箱即用能力。

### 功能点

- 官方模板
- 团队模板
- 模板分类
- 模板预览
- 按模板创建文档
- 将现有文档保存为模板

### 推荐模板

- 会议纪要
- 需求文档
- 周报
- 项目复盘
- 制度文档
- 培训文档
- 招聘面试记录

### UI 设计

模板中心建议采用卡片流布局：
- 顶部为搜索和分类筛选
- 中部为模板卡片网格
- 卡片展示封面、标题、简介、使用次数

## 5.7 权限与分享

### 功能目标

提供接近飞书文档的灵活权限控制，但首版不做过深的复杂策略。

### 功能点

- 文档所有者
- 查看权限
- 编辑权限
- 管理权限
- 链接分享
- 组织内可见
- 指定成员可见
- 公开可见
- 禁止复制
- 禁止导出
- 分享链接失效时间

### UI 设计

采用右侧抽屉或弹窗形式管理分享与权限，分为两个区域：
- 谁可以访问
- 拿到后能做什么

### 交互规则

- 复制链接时直接展示当前权限状态
- 若文档为私有，生成分享链接时需要用户明确选择权限
- 权限修改应立即生效
- 若访问权限收紧，需要提示可能影响当前外部访问者

## 5.8 搜索

### 功能目标

支持飞书式“记不住标题也能找到”的搜索体验。

### 功能点

- 标题搜索
- 正文全文搜索
- 最近搜索
- 搜索建议
- 按空间筛选
- 按类型筛选
- 按创建者筛选
- 按更新时间排序

### UI 设计

- 顶部全局搜索框固定存在
- 搜索结果页分为筛选栏和结果列表
- 每条结果展示标题、命中摘要、位置、更新时间

### 交互规则

- 输入时显示联想建议
- 搜索结果高亮命中词
- 无权限内容不能进入结果集

## 5.9 历史版本

### 功能目标

支持文档回溯和恢复，解决误修改、误删除和内容回退问题。

### 功能点

- 自动版本快照
- 手动保存版本
- 版本列表
- 版本备注
- 版本预览
- 恢复历史版本

### UI 设计

建议采用右侧历史版本面板：
- 时间线展示版本
- 每个版本展示操作人、时间、版本备注
- 点击后进入只读预览

### 交互规则

- 自动保存不一定每次都生成正式版本
- 重要节点可手动创建命名版本
- 恢复版本前需二次确认

## 5.10 导入导出

### 功能点

- 导入 Markdown
- 导入 Docx
- 导出 Markdown
- 导出 PDF
- 导出 Docx

### UI 设计

- 导入作为全局入口放在工作台和文档列表页
- 导出入口放在文档详情页右上角更多操作中

### 交互规则

- 导入后需要生成一篇新文档
- 导出前允许选择格式
- 若文档含不兼容块，需提示可能格式损失

## 5.11 收藏、最近访问、回收站

### 收藏

- 用户可收藏高频文档
- 收藏文档显示在独立导航页

### 最近访问

- 记录最近打开的文档
- 按时间倒序展示

### 回收站

- 删除文档进入回收站
- 支持恢复
- 支持彻底删除

### UI 设计

三者都建议采用统一的列表布局，降低理解成本。

## 6. UI 设计详细说明

## 6.1 整体设计原则

- 风格参考现代企业文档工具
- 保持清爽、轻量、专业
- 减少传统 Office 式高密度工具栏
- 强调内容本身而非复杂操作面板
- 使用稳定的布局层级提升可学习性

## 6.2 页面框架

### 全局框架

- 顶部：全局搜索、新建按钮、用户菜单
- 左侧：主导航
- 主内容区：页面内容

### 文档详情框架

- 顶部：标题区、保存状态、收藏、分享、更多
- 左侧：目录树或文档大纲
- 中间：编辑区
- 右侧：上下文面板

## 6.3 关键页面 UI 说明

### 页面一：工作台

目标：
- 让用户快速找到文档和常用入口

模块：
- 全局搜索
- 新建按钮
- 最近访问
- 收藏
- 常用模板
- 空间入口

### 页面二：空间/知识库页

目标：
- 让用户进行文档管理和结构化浏览

模块：
- 空间名称
- 成员信息
- 目录树
- 列表视图/卡片视图切换
- 排序筛选
- 新建文档/文件夹

### 页面三：文档详情页

目标：
- 让用户专注文档创作

模块：
- 标题输入
- 编辑区
- 大纲
- 保存状态
- 权限入口
- 分享入口
- 历史版本入口

### 页面四：模板中心

目标：
- 帮助用户快速生成结构完整的文档

模块：
- 分类筛选

- 模板卡片
- 预览弹窗

### 页面五：分享与权限面板

目标：
- 用低理解成本完成权限配置

模块：

## 6.4 视觉层建议

### 颜色

- 主色建议使用偏中性的企业产品色
- 避免过重或娱乐化配色
- 功能提示色要清晰区分成功、警告、错误

### 字体

- 中文场景优先保证可读性
- 标题、正文、辅助信息要有明确层级

### 留白

- 编辑区周边留白要足够
- 列表页密度中等，不宜太挤

### 图标

- 图标风格统一
- 文档类型图标需可扩展

## 7. 交互策略

## 7.1 编辑策略

由于 V1 不做协同，建议采用编辑锁策略：
- 文档可多人查看
- 同一时间仅允许一人进入编辑态
- 他人进入时默认只读
- 若当前编辑者离开或超时，锁自动释放

## 7.2 自动保存策略

- 用户有编辑行为后自动保存
- 保存中显示状态
- 保存成功后更新时间
- 网络异常时给出明显提示

## 7.3 新建策略

- 支持从空白新建
- 支持从模板新建
- 支持在当前目录直接新建
- 新文档默认创建在当前空间和当前目录下

## 7.4 删除策略

- 默认软删除
- 回收站保留一定时间
- 恢复后回到原目录

## 8. 文档类型扩展设计

虽然首版只有 `doc`，但设计中必须明确以下机制：

### 8.1 类型字段

所有文档对象必须带有：
- `document_type`

首版值：
- `doc`

预留值：
- `sheet`
- `board`
- `form`
- `database`

### 8.2 类型能力分层

所有类型共享：
- 空间归属
- 权限
- 分享
- 搜索
- 收藏
- 回收站
- 版本

不同类型独有：
- 编辑器
- 渲染器
- 内容 schema
- 工具栏
- 模板类型

### 8.2.1 `doc` 类型内容模型

`doc` 类型建议采用结构化文档树：
- 块级节点：标题、段落、列表、任务列表、引用、代码块、图片、表格、分割线
- 行内节点或样式：加粗、斜体、下划线、删除线、链接、行内代码
- 存储格式：JSON
- 展示形式：连续长文档
- 交互形式：支持块级插入与块级编辑

该设计可以同时满足：
- 飞书式文档观感
- 后续模板扩展
- 后续版本 diff
- 后续评论、审阅、协同能力接入

### 8.3 UI 预留设计

- 新建按钮的交互设计应预留“新建不同类型文档”菜单
- 文档列表中图标和筛选器应预留类型展示
- 搜索结果中预留文档类型标签

## 10. 设计结论

本产品的最佳首版定义不是“简化版协作文档”，而是：

“一款功能完整、结构清晰、UI 现代、支持未来多文档类型扩展的在线云文档系统。”

产品设计上可以充分参考飞书云文档的能力框架，但在首版范围控制上应明确：
- 先把文档做好
- 先把管理和体验做完整
- 暂时不碰协同复杂度
- 底层为后续多类型扩展留好口子
