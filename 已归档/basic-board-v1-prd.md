# CloudDoc 基础画板 V1 PRD

版本：V1.0
日期：2026-04-22
状态：待评审
范围：独立画板文档类型，不包含协作，不包含嵌入普通文档

## 1. 背景

CloudDoc 当前已经支持普通文档、PDF、DOCX 导入、文件夹树、权限、分享、评论、MCP 等基础能力。后续需要增加「画板」能力，但完整飞书式文档内嵌画板、多人协作、AI 生成和开放接口范围过大，不适合第一版一次完成。

本 PRD 将第一版画板能力收敛为「独立基础画板」：

- 画板是一个独立文档类型。
- 画板可以出现在文件夹树和文档列表中。
- 用户可以新建、打开、编辑、保存基础画板。
- 第一版只支持单人编辑。
- 第一版不支持把画板嵌入普通文档。

## 2. 产品目标

### 2.1 核心目标

- 建立 CloudDoc 的画板文件类型。
- 提供一个可用的基础画板编辑器。
- 支持基础流程图绘制。
- 支持保存、重新打开、继续编辑。
- 保持与现有文件夹、权限、删除确认、最近访问等基础文档系统一致。

### 2.2 非目标

V1 明确不做：

- 不做多人实时协作。
- 不做普通文档内嵌画板块。
- 不做画板对象评论。
- 不做 AI 生成画板。
- 不做 MCP / Open API 画板读写。
- 不做复杂模板市场。
- 不做 UML 全量图形库。
- 不做复杂白板能力，例如自由画笔、便利贴、激光笔、会议协作。
- 不做 SVG / PDF 导出。
- 不做移动端深度适配，只保证基础可查看。

## 3. 用户角色

### 3.1 个人用户

使用基础画板整理流程、结构图、简单架构图。

### 3.2 团队成员

在团队空间中新建画板文件，用于需求说明、业务流程梳理、技术方案草图。

### 3.3 管理员

管理画板文件的归属、删除、权限和空间位置。

## 4. 核心用户故事

- 作为用户，我可以在文档库中新建一个画板文件。
- 作为用户，我可以在文件夹中创建画板，让画板和普通文档、PDF 一起被管理。
- 作为用户，我可以打开画板并绘制基础流程图。
- 作为用户，我可以添加矩形、圆形、菱形、文本和连接线。
- 作为用户，我可以移动、缩放、删除画板中的对象。
- 作为用户，我可以保存画板，刷新后内容不丢失。
- 作为用户，我可以在只读权限下查看画板，但不能编辑。

## 5. 信息架构

### 5.1 文档类型

新增文档类型：

- `document_type = board`

文件树显示：

- 普通文档：文档图标
- PDF：PDF 图标
- 画板：画板图标

### 5.2 入口

第一版入口只放在文档库和文件夹页面：

- 左侧「+ 新建文档」弹窗中增加「画板」类型。
- 文件夹工作区「新建文档」弹窗中增加「画板」类型。
- 画板创建时可以选择空间和文件夹位置。

第一版不提供：

- 普通文档编辑器 `/画板` 命令。
- 普通文档块工具菜单中的画板入口。
- 普通文档内嵌画板预览。

### 5.3 页面路由

建议路由：

- 画板查看 / 编辑：`/boards/{boardId}`
- 或复用现有文档详情路由：`/docs/{docId}`，内部根据 `document_type = board` 渲染画板编辑器。

推荐 V1 复用 `/docs/{docId}`：

- 减少权限、最近访问、分享、面包屑、文件夹定位的重复开发。
- 后续如果画板复杂度提高，再拆独立 `/boards/{boardId}` 页面。

## 6. 页面样式

### 6.1 文档库列表样式

画板在文档库中表现为一种文件：

- 标题显示画板名称。
- 类型显示 `board` 或「画板」。
- 图标使用简洁线框画板图标。
- 支持移动、删除、重命名。
- 删除必须弹确认。

### 6.2 画板页面布局

画板页面采用专业工具台布局：

- 左侧保留现有文档树。
- 中间是画板编辑区域。
- 顶部是画板工具栏。
- 右侧属性面板 V1 可以先不做常驻，只在选中对象时显示简化浮层。

推荐布局：

```text
┌────────────────────────────────────────────────────────────┐
│ 顶部栏：返回 / 标题 / 保存状态 / 只读-编辑模式 / 保存       │
├──────────────┬─────────────────────────────────────────────┤
│ 文档树       │ 工具栏                                      │
│              ├─────────────────────────────────────────────┤
│              │                                             │
│              │                 画布区域                    │
│              │                                             │
│              └─────────────────────────────────────────────┤
│              │ 缩放控制 / 适应屏幕                         │
└──────────────┴─────────────────────────────────────────────┘
```

### 6.3 视觉风格

延续 CloudDoc 当前风格：

- 专业、紧凑、少圆角或无圆角。
- 工具栏高度控制在 40px 到 48px。
- 工具按钮 32px 左右，图标优先，文字作为 tooltip。
- 画布背景为浅灰网格。
- 选中对象使用蓝色描边和控制点。
- 辅助线使用细蓝线。
- 不使用重阴影、大卡片、大圆角。

## 7. 功能需求

### 7.1 新建画板

功能：

- 新建弹窗中选择「画板」。
- 输入标题。
- 选择创建位置。
- 创建后进入画板编辑页面。

规则：

- 默认标题为「未命名画板」。
- 默认位置为当前文件夹。
- 画板遵循现有空间和文件夹权限。
- 创建后出现在文件夹树和当前目录列表中。

### 7.2 打开画板

功能：

- 点击文件树中的画板打开。
- 点击列表中的画板打开。
- 最近访问中可以打开画板。

规则：

- 有编辑权限时默认进入编辑模式。
- 只有阅读权限时进入只读模式。
- 如果后端接口失败，页面必须显示错误提示，不能假装为空。

### 7.3 顶部栏

显示内容：

- 返回文件夹 / 文档库。
- 画板标题。
- 更新时间。
- 保存状态：已保存、保存中、保存失败。
- 模式切换：只读 / 编辑。
- 保存按钮。

功能：

- 标题可重命名。
- 编辑模式下修改自动保存。
- 保存失败要提示。
- 切换到只读前，如果存在未保存变更，必须先保存。

### 7.4 画布

基础能力：

- 平移画布。
- 缩放画布。
- 适应屏幕。
- 画布网格背景。
- 鼠标滚轮缩放或滚动，具体交互实现时统一设计。

缩放范围：

- 最小 25%。
- 最大 400%。
- 默认 100%。

### 7.5 工具栏

V1 工具：

- 选择工具。
- 手型拖动画布。
- 文本。
- 矩形。
- 圆角矩形。
- 圆形 / 椭圆。
- 菱形。
- 连接线。
- 删除。
- 撤销。
- 重做。

工具交互：

- 鼠标悬浮显示 tooltip。
- 当前工具高亮。
- 使用选择工具点击对象后进入选中态。
- 使用图形工具点击画布后创建对象。
- 使用连接线工具从一个节点拖到另一个节点创建连接。

### 7.6 图形对象

V1 对象类型：

- `text`
- `rectangle`
- `round_rectangle`
- `ellipse`
- `diamond`
- `connector`

对象基础属性：

- `id`
- `type`
- `x`
- `y`
- `width`
- `height`
- `text`
- `style`
- `zIndex`

样式属性：

- 填充色。
- 边框色。
- 边框宽度。
- 字体大小。
- 字体颜色。

V1 属性编辑可以简化为：

- 文本编辑。
- 填充色选择。
- 边框色选择。

### 7.7 连接线

基础能力：

- 连接两个图形对象。
- 支持箭头终点。
- 拖动图形后连接线跟随。
- 删除图形时关联连接线一并删除。

V1 不做：

- 自动避让。
- 多段路径手动编辑。
- 连接线上添加标签。
- 起点箭头。

### 7.8 对象选择与编辑

功能：

- 单击选中对象。
- 拖拽移动对象。
- 拖拽控制点缩放对象。
- 双击文本或图形进入文字编辑。
- Delete / Backspace 删除选中对象。
- Cmd/Ctrl + C 复制。
- Cmd/Ctrl + V 粘贴。
- Cmd/Ctrl + Z 撤销。
- Cmd/Ctrl + Shift + Z 重做。

V1 可暂不做：

- 框选多选。
- 对齐分布。
- 组合。
- 图层置顶置底。

### 7.9 保存

保存机制：

- 编辑后自动保存。
- 顶部保留手动保存按钮。
- 自动保存节流，例如 1 秒到 2 秒。
- 保存失败后显示明显提示。

保存内容：

- 画板标题。
- 画布视图状态。
- 所有节点。
- 所有连接线。

### 7.10 删除

删除场景：

- 删除画板文件。
- 删除画板对象。

规则：

- 删除画板文件必须弹确认。
- 删除画板对象 V1 可以不弹确认，但必须支持撤销。
- 如果删除画板文件，进入回收站或现有删除流程。

## 8. 权限规则

V1 复用现有文档权限。

### 8.1 可读

- 可以打开画板。
- 可以缩放和平移查看。
- 不能新增、修改、删除对象。
- 不能保存。

### 8.2 可编辑

- 可以修改画板内容。
- 可以重命名画板。
- 可以保存。

### 8.3 可管理

- 可以删除画板文件。
- 可以移动画板文件。
- 可以调整权限。

## 9. 数据设计

### 9.1 documents 表

新增支持：

- `document_type = board`

画板仍作为文档系统中的一个文件对象：

- 使用 `documents.id` 作为画板文件 ID。
- 使用 `documents.title` 作为画板标题。
- 使用 `documents.folder_id` 管理位置。
- 使用 `documents.visibility` 管理基础可见性。

### 9.2 board_contents 表或 DocumentContent

V1 有两个可选方案：

#### 方案 A：复用 DocumentContent

优点：

- 版本、保存、详情接口可复用。
- 实现快。

缺点：

- 画板 JSON 和普通文档 JSON 混在同一套内容表中。

#### 方案 B：新增 board_contents 表

优点：

- 画板数据独立，后续扩展节点、协作、API 更清晰。

缺点：

- V1 开发量更大。

V1 推荐：

- 采用方案 A，先复用 `DocumentContent.content_json`。
- 但 `content_json.type` 必须明确为 `board`，和普通文档区分。
- 后续 V2/V3 如果需要对象级版本或协作，再拆 `boards` / `board_nodes`。

### 9.3 content_json 格式

示例：

```json
{
  "type": "board",
  "version": 1,
  "viewport": {
    "x": 0,
    "y": 0,
    "zoom": 1
  },
  "nodes": [
    {
      "id": "node_1",
      "type": "round_rectangle",
      "x": 120,
      "y": 100,
      "width": 160,
      "height": 56,
      "text": "开始",
      "style": {
        "fill": "#eff6ff",
        "stroke": "#2563eb",
        "strokeWidth": 1,
        "fontSize": 14,
        "color": "#0f172a"
      },
      "zIndex": 1
    }
  ],
  "connectors": [
    {
      "id": "connector_1",
      "from": "node_1",
      "to": "node_2",
      "points": [],
      "style": {
        "stroke": "#475569",
        "strokeWidth": 1,
        "arrowEnd": true
      },
      "zIndex": 0
    }
  ]
}
```

## 10. API 需求

V1 可以复用现有文档 API，但需要支持 `board` 类型。

### 10.1 创建画板

可选：

- 扩展 `POST /api/documents`
- 或新增 `POST /api/boards`

推荐 V1：

- 扩展 `POST /api/documents` 支持 `document_type = board`。

创建后默认内容：

```json
{
  "type": "board",
  "version": 1,
  "viewport": { "x": 0, "y": 0, "zoom": 1 },
  "nodes": [],
  "connectors": []
}
```

### 10.2 获取画板

复用：

- `GET /api/documents/{id}`

要求：

- 返回 `document_type = board`。
- 返回 `content.content_json.type = board`。

### 10.3 保存画板

复用：

- `PUT /api/documents/{id}/content`

要求：

- 后端校验当前文档类型是 `board`。
- 后端校验内容 JSON 是合法画板格式。
- 只允许有编辑权限的用户保存。

## 11. 前端页面需求

### 11.1 文件夹工作区

- 新建弹窗增加「画板」选项。
- 文件列表和文件树支持画板图标。
- 点击画板进入画板页面。

### 11.2 文档详情渲染分流

在 `/docs/{docId}` 中：

- `document_type = doc`：渲染普通文档编辑器。
- `document_type = pdf`：渲染 PDF 查看逻辑。
- `document_type = board`：渲染画板编辑器。

### 11.3 画板编辑器组件

建议组件：

- `BoardPage`
- `BoardToolbar`
- `BoardCanvas`
- `BoardObjectLayer`
- `BoardSelectionOverlay`
- `BoardInspector`

V1 可以先不做完整 `BoardInspector`，仅做简单浮层。

- 用户能打开画板页面。

## 15. 风险

### 15.1 编辑器复杂度失控

控制方式：

- V1 不做协作。
- V1 不做普通文档嵌入。
- V1 不做专业图形库。
- V1 不做模板。

### 15.2 数据结构后续迁移

控制方式：

- V1 虽复用 `DocumentContent`，但 `content_json.type` 必须是 `board`。
- 所有画板内容集中在 `content_json.nodes` 和 `content_json.connectors`，避免散乱字段。

### 15.3 权限越权

控制方式：

- 画板详情和保存走现有文档权限服务。
- 分享页面只读。
- 私有画板不能通过原链接越权访问。

## 16. 结论

基础画板 V1 的目标是建立「独立画板文件」能力，而不是完整复刻飞书文档内嵌画板。

本期只做：

- 独立画板文档类型。
- 基础画板编辑器。
- 基础流程图对象。
- 保存和重新打开。
- 权限只读 / 编辑控制。

本期坚决不做：

- 协作。
- 普通文档嵌入。
- AI。
- MCP / Open API。
- 高级白板。

这个范围足够小，可以尽快进入开发和自动化测试，同时为后续专业流程图、文档内嵌、协作和 AI 留出扩展空间。
